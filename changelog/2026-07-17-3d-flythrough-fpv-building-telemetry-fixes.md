### Added

- DJI Cloud fleet panel now shows a live telemetry HUD for the tracked aircraft: signal strength (from its paired RC/dock), battery, ASL/ALT altitude, distance from home, climb rate, flight speed, and wind — updating in real time over the existing telemetry connection.

### Fixed

- The 3D flythrough camera was both choppy and visibly lagging behind the drone's actual position. It teleported to a new spot only as often as the playback scrubber ticked (a handful of times a second), and an earlier attempt to smooth that out by easing toward each new position made it worse — an eased camera is, by definition, always chasing behind a moving target. It now runs its own real-time clock and computes the mathematically exact position along the route for the current elapsed playback time on every animation frame (~60fps), so it's always precisely where the drone is, with continuous rather than stepped motion. It also now positions the camera as a true first-person view (Mapbox's FreeCameraOptions) at the drone's real altitude, heading, and gimbal pitch, instead of a fixed-angle overhead chase camera.
- Buildings marked with a height via "Budova" never actually rendered as extruded 3D volumes, even in 3D map mode — react-map-gl silently refuses to change a layer's type (flat fill → 3D extrusion) after it's first created, so whichever mode the map happened to be in when a building was first drawn stuck permanently. Both layers are now always present, toggled by opacity instead of type, matching the pattern already used elsewhere in the map (camera frustums).
- The PDF report's numbered waypoint markers only showed a number on the very first marker — every marker after it silently lost its fill/text color, because jsPDF drops the active color after a `text()` call using a custom embedded font unless it's re-applied on every draw. Verified against an actual rendered PDF, not just the mocked test suite.
- Live video streaming always failed with "Spuštění živého přenosu selhalo". The platform's own capacity API returns a random, meaningless UUID as each video's `id` — the video identifier the start/stop endpoints actually validate against has to be assembled from the device's serial number plus the camera's and video's own index fields instead.
